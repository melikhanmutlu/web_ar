Fix: Cumulative Scale Tracking & Material Cache Bust

Problem 1: Scale Tracking
User concern: "After saving scaled model, slider resets to 1.0x. User can keep scaling 10x repeatedly, creating huge models. Can't see total scale from original."

Example:
```
Original: 50cm
User: Scale 2x → Save → 100cm
Slider: Resets to 1.0x ❌

User: Scale 2x again → Save → 200cm
User thinks: "2x twice = 4x total" ✅
Display: Shows 1.0x ❌ (Confusing!)
```

Solution: Cumulative Scale Tracking

Database Changes:
```python
class UserModel:
    original_dimensions = db.Column(db.JSON)  # Original at upload
    cumulative_scale = db.Column(db.Float, default=1.0)  # Total scale
```

Upload Flow:
```python
# At upload (app.py line 1170)
model = UserModel(
    original_dimensions={
        'x': 50.0,
        'y': 60.0,
        'z': 40.0,
        'max': 60.0
    },
    cumulative_scale=1.0  # Initial
)
```

Save Flow:
```python
# At save_modifications (app.py line 2054)
if applied_scale != 1.0:
    current_cumulative = model.cumulative_scale or 1.0
    model.cumulative_scale = current_cumulative * applied_scale
    
# Example:
# Save 1: 1.0 * 2.0 = 2.0x
# Save 2: 2.0 * 1.5 = 3.0x
# Save 3: 3.0 * 0.5 = 1.5x
```

Frontend Display:
```html
<!-- Transform Panel -->
<div class="flex justify-between">
    <span>Total Scale:</span>
    <span id="cumulativeScale">2.5x</span>
</div>
```

```javascript
// Load from backend
const cumulativeScale = {{ cumulative_scale }};
document.getElementById('cumulativeScale').textContent = 
    cumulativeScale.toFixed(2) + 'x';
```

User Experience:
```
Original: 50cm (cumulative: 1.0x)

User: Scale 2x → Save
Current: 100cm (cumulative: 2.0x) ✅
Display: "Total Scale: 2.0x"

User: Scale 1.5x → Save
Current: 150cm (cumulative: 3.0x) ✅
Display: "Total Scale: 3.0x"

User: Scale 0.5x → Save
Current: 75cm (cumulative: 1.5x) ✅
Display: "Total Scale: 1.5x"
```

Benefits:
✅ User sees total scale from original
✅ Prevents accidental huge models
✅ Clear feedback on modifications
✅ Can calculate back to original size

Problem 2: Material Appearance Change After Save

User report: "After saving, material looks washed out. Color same, metalness/roughness same, but appearance different."

Root Cause: Browser cache serving old GLB

Before:
```javascript
// Reload without cache bust
window.location.reload();
// Browser may serve cached GLB!
```

After:
```javascript
// Reload with timestamp
window.location.href = window.location.pathname + '?t=' + Date.now();
// Forces fresh GLB load
```

Why This Happens:
1. User modifies material
2. Save creates new GLB
3. Page reloads
4. Browser cache serves OLD GLB
5. Materials look different (old version)

Solution: Cache Busting
```javascript
// Old URL: /view/model-id
// New URL: /view/model-id?t=1730487123456
// Browser treats as different URL → fresh load
```

Result:
✅ Fresh GLB loaded every time
✅ Materials appear correctly
✅ No washed out appearance
✅ Consistent visual quality

Upload Logic:
```python
# Parse dimensions
original_dims = {
    'x': bounds_data['extents'][0],
    'y': bounds_data['extents'][1],
    'z': bounds_data['extents'][2],
    'max': bounds_data['max']
}

# Create model
model = UserModel(
    original_dimensions=original_dims,
    cumulative_scale=1.0
)
```

Save Logic:
```python
# Update cumulative scale
if 'transform' in modifications and 'scale' in modifications['transform']:
    applied_scale = float(modifications['transform']['scale'])
    if applied_scale != 1.0:
        current_cumulative = model.cumulative_scale or 1.0
        model.cumulative_scale = current_cumulative * applied_scale
        logger.info(f"Updated cumulative scale: {current_cumulative} * {applied_scale} = {model.cumulative_scale}")
```

View Template:
```python
# Pass to template
return render_template('view.html',
    cumulative_scale=model.cumulative_scale or 1.0
)
```

Frontend Display:

    cumulativeScale.toFixed(2) + 'x';
